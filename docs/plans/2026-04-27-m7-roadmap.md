## 1. Track A — Launch-critical (must ship before public launch)

### A1 — Real LLM provider integration (replace M6.5b stub)

**Issue:** M7-A1 (linked below)

**Why:** PDF translation currently outputs `[Pro stub: <model> 将在 M6.5b 接通] <original>` for any LLM model. Free Google + Microsoft work, but the 9 LLM models are user-visible-but-broken. **Public launch with this stub is misleading.**

**Files:**
- Modify: `apps/api/src/translate/dispatch.ts:30-40` — replace stub branch with real call
- Possibly Create: `apps/api/src/translate/llm-providers.ts` — bianxie / openrouter / direct provider clients
- Modify: `apps/api/src/translate/__tests__/dispatch.test.ts` — replace stub assertion with mocked provider
- Modify: `apps/api/wrangler.toml` (if new env vars) — already has `BIANXIE_BASE_URL`

**Approach:**
- Use the existing AI proxy (`apps/api/src/ai/proxy.ts` is OpenAI-compatible)
- For text translation: stream completion and dump to result.text
- For document translation: bianxie call with system prompt to translate chunk
- Consume `web_text_translate_token_monthly` quota for LLM (already wired in M6.3)
- Each LLM call returns `tokens: { input, output }` so quota is accurate

**Acceptance:**
- [ ] `dispatchTranslate("gpt-5.5", text, "auto", "zh-Hans")` returns real translation
- [ ] Token counts populated and consumed against quota
- [ ] Rate limit / 5xx errors surface as `ORPCError("PROVIDER_FAILED")` (M6.5b shape preserved)
- [ ] PDF translation pipeline produces real LLM output (no `[Pro stub:` prefix)

**Pre-conditions:** None (LLM API access already configured via `BIANXIE_API_KEY` secret).

### A2 — Rate limiting per user/IP at api edge

**Issue:** M7-A2

**Why:** No abuse protection. A malicious user (or bug) could burn LLM tokens / R2 quota / D1 writes at scale. M6.x quota system limits per-user usage but doesn't limit RPS.

**Files:**
- Create: `apps/api/src/middleware/rate-limit.ts` — KV-backed sliding window rate limiter
- Modify: `apps/api/src/index.ts` — add middleware to all `/orpc/*` and `/ai/v1/*` routes
- Modify: `apps/api/wrangler.toml` — add KV binding `RATE_LIMIT_KV` (existing `apps/api/src/ai/rate-limit.ts` may already use KV — reuse the binding)
- Create: `apps/api/src/middleware/__tests__/rate-limit.test.ts`

**Approach:**
- Sliding window algorithm: 60 req/min per authenticated user, 30 req/min per IP for anonymous
- Cloudflare Workers KV for state (already used by `ai/rate-limit.ts`)
- 429 response with `Retry-After` header
- Whitelist for internal calls (CI smoke tests, cron self-checks)

**Acceptance:**
- [ ] User exceeding 60 RPM gets 429 + `Retry-After`
- [ ] State decays correctly (sliding window, not fixed)
- [ ] Anonymous IP-based limiting works (X-Forwarded-For from CF)
- [ ] Smoke test still passes (whitelisted)

**Pre-conditions:** `RATE_LIMIT_KV` binding may need creating: `wrangler kv namespace create RATE_LIMIT_KV`.

### A3 — Auto-rollback on deploy smoke test failure

**Issue:** M7-A3

**Why:** Currently if `wrangler deploy` succeeds but smoke test fails, the broken Worker stays live. Manual rollback required. Adding auto-rollback closes the window between bad-deploy and detection.

**Files:**
- Modify: `.github/workflows/deploy-api.yml` — add rollback step
- Modify: `.github/workflows/deploy-web.yml` — add equivalent rollback step (Pages has different rollback mechanism)
- Possibly add: `apps/api/scripts/rollback.sh` — if rollback logic gets complex

**Approach:**
- For Worker: capture deploy id, on smoke failure run `wrangler deploy --version <previous>` (Cloudflare Workers Versions API)
- For Pages: on deploy failure mark deployment "abandoned" via dashboard API
- Always alert on rollback (PostHog event `deploy_rollback` + Sentry breadcrumb)

**Acceptance:**
- [ ] Inject a fault that makes smoke test fail → workflow detects → rolls back → reports failure with previous version live
- [ ] Document rollback verification procedure in DEPLOY-CHECKLIST.md

**Pre-conditions:** None (uses existing wrangler auth).

## 2. Track B — Operational safety (post-launch hardening)

### B1 — `progress_updated_at` column for stuck-job detection

**Issue:** M7-B1

**Why:** M6.12 stuck-sweep uses `created_at < now - 30min` heuristic. False-positive risk: a legitimately long PDF (200 pages, slow LLM) gets killed mid-flight. Tracking last-progress-update is more accurate.

**Files:**
- Modify: `packages/db/src/schema/translate.ts` — add `progressUpdatedAt: integer("progress_updated_at", { mode: "timestamp_ms" })`
- Create: `packages/db/drizzle/0007_<name>.sql` — migration (run `pnpm --filter @getu/db generate`)
- Modify: `apps/api/src/queue/translate-document.ts` — every progress write also updates `progressUpdatedAt`
- Modify: `apps/api/src/scheduled/translation-stuck-sweep.ts` — change WHERE from `created_at` to `progress_updated_at`
- Update: `apps/api/src/scheduled/__tests__/translation-stuck-sweep.test.ts`

**Acceptance:**
- [ ] Migration adds nullable column safely
- [ ] M6.12 stuck-sweep uses new column; tests cover both populated and NULL (legacy rows) cases
- [ ] Threshold can stay at 30min (or relax to 60min if too aggressive)

**Pre-conditions:** None.

### B2 — Logger sampling / level filter for PostHog forwarding

**Issue:** M7-B2

**Why:** `apps/api/src/analytics/logger.ts` forwards every `info/warn/error` call to PostHog as `internal_log`. Currently 1 call site (text.ts), but if logger usage spreads, PostHog 1M events/month free tier is at risk.

**Files:**
- Modify: `apps/api/src/analytics/logger.ts` — add level filter (only `error` forwards by default; `warn` + `info` console-only) + per-event sampling rate config
- Modify: `apps/api/src/analytics/__tests__/logger.test.ts`

**Acceptance:**
- [ ] `logger.info()` does NOT forward to PostHog (console only)
- [ ] `logger.warn()` does NOT forward by default; can be opted-in via `{ forward: true }` 4th arg
- [ ] `logger.error()` always forwards
- [ ] Test asserts the forwarding gate

**Pre-conditions:** None.

### B3 — Spend cap monitoring + Slack alerts

**Issue:** M7-B3

**Why:** Cost surprise risk. If LLM tokens / R2 ops / D1 writes spike (abusive user, bug), no one knows until end-of-month bill. Need daily spend monitor that alerts above threshold.

**Files:**
- Create: `apps/api/src/scheduled/spend-monitor.ts` — daily aggregate over `usage_log`
- Create: `apps/api/src/scheduled/__tests__/spend-monitor.test.ts`
- Modify: `apps/api/src/worker.ts` scheduled handler — add new cron task
- Modify: `apps/api/wrangler.toml` — secret `SLACK_WEBHOOK_URL` for alerts
- Modify: `apps/api/DEPLOY-CHECKLIST.md` — list new secret

**Approach:**
- Daily 03:00 UTC (existing cron) sums `usage_log.amount` per `bucket` over last 24h
- If any bucket exceeds threshold (per env config), POST to Slack webhook
- Thresholds in env: `SPEND_ALERT_PDF_PAGES_PER_DAY=200`, `SPEND_ALERT_LLM_TOKENS_PER_DAY=500000`, etc

**Acceptance:**
- [ ] Aggregation correct over usage_log
- [ ] Slack message renders cleanly with bucket name + threshold + actual value
- [ ] Test asserts no alert when below threshold + alert when above

**Pre-conditions:** User configures Slack webhook in their workspace.

### B4 — Real Sentry alert routing

**Issue:** M7-B4 (ops-only, no code)

**Why:** Sentry receives errors but no one is paged. Configure routing.

**Action:**

- Update runbook with the actual rule URLs

**Acceptance:** First test error fires email/Slack within 5 min.

**Pre-conditions:** None (Sentry account set up in M6.13).

### B5 — PostHog dashboards + Insights

**Issue:** M7-B5 (ops-only, no code)

**Why:** Events arrive (M6.13) but nobody looks at them. Build the standing dashboard.

**Insights to create:**
1. DAU split by free/pro
2. Translate funnel: visit → translate click → upgrade-modal-shown → checkout-started

**Acceptance:** Dashboard linked in `runbook-translation-incidents.md`. First weekly check shows usable data.

**Pre-conditions:** Real users producing events (live traffic).
